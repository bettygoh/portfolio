Interface foundations for ADDX.

Design System 2.0 establishes a cohesive, scalable framework to improve consistency across the platform — defining core UI components, interaction patterns and visual guidelines to streamline design and development.

RoleProduct Designer (1 of 4)
Team4 Product Designers · 3 Engineers
TimelineNov 2023 – Apr 2024
ToolsFigma
The problem

Design System 1.0 was holding the team back.

Poorly maintained and under-documented, the old system created an inconsistent experience across web and app — and quietly slowed everyone down.

1

Lack of consistency

Poor maintenance of Design System 1.0 — inconsistent and not properly defined — led to a disjointed experience on web and app.

2

No proper guidelines

The system wasn't properly documented, which created friction every time someone tried to use it.

3

Not aligned to the brand

Gaps in colours and typography across app and web, and a drift from the updated brand style.

4

More front-end errors

With no centralised guideline, engineers wrote custom code — and designers spent more time debugging avoidable visual errors.

The goal

Rebuild the system so it actually gets used.

Three goals shaped the work — closing the brand gap, getting closer to code, and proving the system's return on investment.

1

Bridge the gap with the brand

Rebuild the system with consistency and usage guidelines, so designers and engineers deliver a consistent experience on app and web.

2

Better integration with code

Collaborate with engineers on the implementation and maintenance of the components.

3

Increase ROI of the system

Achieve time and cost savings across design, development and testing.

Prioritisation

Foundations first.

We prioritised foundational elements first — starting with text styles, colours, buttons and text inputs — to establish consistency before refining individual components.

My scope

Buttons, text links, badges, dividers & file upload.

As one of four designers on Design System 2.0, my scope focused on revamping Buttons, Text Links, Badges (Tags), Dividers and the File Upload component. I started with Buttons — collecting existing screens to audit inconsistencies, and researching how other platforms design their buttons to inform best practices.

Insights

Auditing the existing buttons.

From the audit of existing buttons, key inconsistencies emerged across both app and web.

1

Text style & hierarchy issues

  • Web used sentence case mostly, with some uppercase exceptions; app used uppercase with extra-bold font — making the two platforms feel disconnected.
  • App and web each had their own button styles, with different hierarchy variations.
2

Missing & platform-specific buttons

  • Web had a "Danger" button, but the app didn't — no clear destructive action on mobile.
  • App had an "Action Done" button, but web didn't — completed actions were acknowledged on mobile but not on web.
Solution

A unified, scalable, accessible button framework.

1

Consistent button variants

  • Primary — the main call-to-action.
  • Secondary — supporting, lower-emphasis actions.
  • Tertiary — minimal styling for less prominent actions.
  • Danger — destructive actions (now added to mobile).
2

Aligned text & typography

  • Standardised button text to sentence case across web and app.
  • Standardised font weight and size for labels to improve readability.
3

A button style guide

  • Defined a single source of truth for button styling in Figma.
  • Designed and documented hover, focus, disabled and active states.
Impact

~10% lift in designer efficiency.

An estimated 10% improvement in designer efficiency through the adoption of more discoverable and more powerful components.

What I learnt

A design system is more than visuals — it lives or dies on cross-team alignment.

Cross-team collaboration and stakeholder buy-in are essential for a successful design system. It requires alignment between designers, developers and product managers. Partnering closely with engineers helped bridge the gap between design and implementation — ensuring real adoption rather than just documentation.
